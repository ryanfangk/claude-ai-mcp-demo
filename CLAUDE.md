# Project

Next.js 16 + Payload CMS 3.x demo. PostgreSQL (local via docker-compose).
Package manager: pnpm. Dev server: `pnpm dev`.

## Codebase

```
app/(frontend)/      → Public-facing pages and layout
app/(payload)/       → Payload admin panel, REST/GraphQL routes (auto-generated, don't edit)
collections/         → Payload collection configs, access control, hooks
lib/                 → Shared utilities (MCP tools, WorkOS OAuth, helpers)
payload.config.ts    → Payload CMS configuration (root)
payload-types.ts     → Auto-generated TypeScript types (don't edit, run pnpm generate:types)
```

# Workflow

## Regenerate Payload artifacts after collection / config changes

Whenever you edit a collection, hook, custom component reference, or
`payload.config.ts` in a way that affects the admin panel's component
graph or the generated types, regenerate the matching artifact in the
SAME commit as the code change:

| What changed | Run | Output (commit it) |
| --- | --- | --- |
| Fields, defaultColumns, useAsTitle, admin components, custom views | `pnpm generate:importmap` | `app/(payload)/admin/importMap.js` |
| Collection field shape (added/removed/renamed) or relationship | `pnpm generate:types` | `payload-types.ts` |
| Schema drift from collection edits (columns, indexes, FKs) | `pnpm migrate:create <descriptive_name>` then `pnpm migrate` | `migrations/<timestamp>_<name>.{ts,json}` + updated `migrations/index.ts` |

Skipping `generate:importmap` is the specific failure mode that produces
`getFromImportMap: PayloadComponent not found in importMap` at runtime in
the admin panel — the importMap is consulted on every server render and
silently goes stale when a collection's components shift.

Skipping `generate:types` produces TS errors in any code that touches
the changed collection.

Skipping the migration regen leaves the dev DB right (drizzle push
covers it) but ships broken migrations to anyone else.

`app/(payload)/admin/importMap.js` is the ONE file inside `app/(payload)/`
that's regenerated by an explicit command and intended to be committed.
Everything else in that route group is generated at build time — don't edit.

## Environment variables

- When adding a new env var to `.env.local`, also add the non-sensitive version to `.env.example` (empty value or placeholder).

# Code Quality

- Prefer correct, complete implementations over minimal ones.
- Use appropriate data structures and algorithms — don't brute-force what has a known better solution.
- When fixing a bug, fix the root cause, not the symptom.
- If something I asked for requires error handling or validation to work reliably, include it without asking.
